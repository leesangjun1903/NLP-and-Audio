## 1. 핵심 주장과 주요 기여 요약

### 🎯 핵심 주장

> **"장기 사용자 메모리 활용은 시스템 수준의 수동 검색(Passive Retrieval)에서 에이전트 고유의 능동적 메모리 탐색(Active Memory Navigation)으로 전환되어야 한다."**

기존 메모리 시스템들은 미리 선정된 증거를 에이전트에게 전달하는 수동적 검색 인터페이스에 의존합니다. 이로 인해 에이전트는 검색된 맥락이 불완전할 때 추가 증거를 탐색할 능력이 제한됩니다. NapMem은 장기 사용자 메모리를 **구조화된 행동 공간(Structured Action Space)**으로 재정의하여, 에이전트가 쿼리와 중간 증거에 따라 메모리 활용을 스스로 결정하도록 학습시킵니다.

### 📋 주요 기여 (4가지)

| # | 기여 내용 |
|---|-----------|
| 1 | 장기 사용자 메모리 활용을 **능동적 메모리 탐색**으로 재정식화 — 메모리를 수동적으로 검색된 맥락이 아닌 에이전트가 학습하여 사용하는 구조화된 행동 공간으로 전환 |
| 2 | **NapMem 프레임워크** 도입 — 다중 세분도 메모리 피라미드 구축 및 원시 대화, 메모리 레코드, 주제 트랙, 사용자 프로필을 도구로 노출하여 제어 가능한 메모리 접근 실현 |
| 3 | **강화학습(GRPO)** 기반 메모리 탐색 정책 학습 — 메모리 도구 트래젝토리에 대한 강화학습으로 에이전트 훈련 |
| 4 | **6개 벤치마크** 평가 — 메모리 집약적 과제 및 비메모리 과제에서 과제 성능, 일반화, 효율성, 도구 사용 행동 분석 |

## 2. 해결하고자 하는 문제, 제안 방법, 모델 구조, 성능 향상 및 한계

### 2.1 해결하고자 하는 문제

**문제 1: 수동 검색의 근본적 한계**

```
[사용자 쿼리] → [시스템이 미리 검색] → [선택된 컨텍스트 전달] → [에이전트 응답]
```

수동 검색 방식에서는 시스템이 미리 검색한 증거만 에이전트에게 전달됩니다. 검색된 증거가 불완전하거나 관련성이 낮으면, 에이전트는 추가 탐색 없이 오답을 생성할 수 있습니다.

**문제 2: 이질적인 메모리 요구사항**

사용자 쿼리에 따라 필요한 메모리 유형이 다릅니다:
- 사실 기반 질문 → 정확한 원시 대화 증거 필요
- 선호도 관련 질문 → 상위 수준의 추상화된 프로필 필요
- 크로스-세션 질문 → 주제별 집계 정보 필요

단일 고정 메모리 표현이나 접근 전략은 이 다양성에 일반화되기 어렵습니다.

**문제 3: 에이전트의 메모리 활용 통제권 부재**

기존 연구들은 메모리 접근을 시스템 수준의 검색 기능이나 사전 설계된 파이프라인으로 처리하여, 에이전트가 메모리를 어떻게 활용할지에 대한 통제권이 거의 없습니다.

### 2.2 제안하는 방법

#### 2.2.1 다중 세분도 메모리 피라미드 (Multi-Granularity Memory Pyramid)

NapMem은 사용자 상호작용 이력을 4계층 피라미드로 구조화합니다:

```
┌─────────────────────────────┐
│      User Profile (L4)      │  ← 전역 사용자 요약, 안정적 속성
├─────────────────────────────┤
│     Topic Tracks (L3)       │  ← 크로스-세션 주제별 집계
├─────────────────────────────┤
│    Memory Records (L2)      │  ← 압축된 사실/이벤트/지시/선호도
├─────────────────────────────┤
│  Raw Conversations (L1)     │  ← 원시 대화 증거
└─────────────────────────────┘
```

**각 계층 세부사항:**

| 계층 | 내용 | 저장 형식 | 역할 |
|------|------|-----------|------|
| Raw Conversations (L1) | 원시 메시지, 화자, 타임스탬프, 고유 ID | JSONL + 벡터/키워드 인덱스 | 최고 충실도 증거, 세부 검증 |
| Memory Records (L2) | 사실(fact), 이벤트(event), 지시(instruction), 선호도(preference) | JSONL + 하이브리드 검색 | 기본 의미 단위, 압축된 사용자 정보 |
| Topic Tracks (L3) | 주제별 메타데이터, 요약, 서술, 증거 링크 | Markdown 파일 | 중간 범위 추상화, 크로스-세션 맥락 |
| User Profile (L4) | 안정적 속성, 장기 선호도, 상호작용 패턴 | Markdown 파일 | 전역 사용자 이해, 개인화 |

**피라미드 증분 구축 알고리즘 (Algorithm 1):**

$$\mathcal{M} = \{\mathcal{C}, \mathcal{R}, \mathcal{T}, P\}$$

여기서 $\mathcal{C}$는 원시 대화 저장소, $\mathcal{R}$은 레코드 풀, $\mathcal{T}$는 주제 트랙 집합, $P$는 사용자 프로필입니다. 구축은 하위 계층에서 상위 계층으로 증분적으로 전파됩니다.

#### 2.2.2 도구 기반 메모리 탐색

**공식화:**

사용자 쿼리 $q$와 메모리 피라미드 $\mathcal{M}$이 주어지면, 에이전트는 최종 답변 생성 전 순차적으로 메모리와 상호작용합니다. 각 단계 $t$에서 에이전트는 메모리 도구를 호출하거나 탐색을 종료하고 답변합니다:

$$a_t \sim \pi_\theta(\cdot \mid q, \mathcal{M}, a_{ < t}, o_{ < t})$$

여기서 $a_t$는 다음 행동, $o_t$는 메모리 도구 호출 시 반환되는 관측값입니다.

**메모리 사용 트래젝토리:**

$$\tau = (a_1, o_1, \ldots, a_k, o_k, y)$$

여기서 $y$는 최종 답변입니다.

**5가지 메모리 도구:**

| 도구 | 입력 | 기능 |
|------|------|------|
| `search_records` | 자유 텍스트 쿼리 | 하이브리드 RRF 검색으로 상위 5개 메모리 레코드 반환 |
| `search_conversation` | 자유 텍스트 쿼리 | 하이브리드 RRF 검색으로 상위 5개 대화 스니펫 반환 |
| `get_records` | 레코드 ID 목록 | 레코드 ID로 정확한 메모리 레코드 조회 |
| `get_conversation` | 메시지 ID 목록 | 메시지 ID로 정확한 대화 메시지 조회 |
| `read_file` | 파일명 | 주제 트랙 파일 또는 profile.md 읽기 |

#### 2.2.3 GRPO를 이용한 메모리 탐색 학습

**보상 함수 (Rule-based Reward):**

세 가지 이진 기준으로 보상을 정의합니다:
- $F$: 형식 유효성 (Format validity)
- $C$: 답변 정확성 (Answer correctness)
- $U$: 메모리 도구 사용 여부 (Memory-tool usage)

$$r(\tau) = \begin{cases} -1, & F = 0 \\ 1, & F = 1,\ C = 1,\ U = 1 \\ 0, & F = 1,\ C = 1,\ U = 0 \\ -0.5, & F = 1,\ C = 0,\ U = 1 \\ -1, & F = 1,\ C = 0,\ U = 0 \end{cases}$$

**그룹 상대적 이점 (Group-Relative Advantage):**

각 쿼리에 대해 샘플링된 트래젝토리 그룹 $\mathcal{G}$에서, 트래젝토리 $\tau_i$의 보상 $r_i = r(\tau_i)$로 그룹 상대적 이점을 계산합니다:

$$A_i = r_i - \frac{1}{|\mathcal{G}|} \sum_{j \in \mathcal{G}} r_j$$

**클리핑된 GRPO 목적 함수:**

$$\mathcal{J}(\theta) = \mathbb{E}\left[\frac{1}{|\mathcal{G}|}\sum_{i=1}^{|\mathcal{G}|} \frac{1}{T_i} \sum_{t=1}^{T_i} \ell_{i,t}(\theta) - \beta D_{\text{KL}}(\pi_\theta \| \pi_{\text{ref}})\right]$$

여기서:

$$\ell_{i,t}(\theta) = \min\left(\rho_{i,t} A_i,\ \text{clip}(\rho_{i,t},\ 1-\epsilon,\ 1+\epsilon) A_i\right)$$

$\rho_{i,t}$는 토큰 수준 중요도 비율(importance ratio), $T_i$는 트래젝토리 $\tau_i$의 생성 토큰 수입니다. 트래젝토리 수준의 이점은 메모리 도구 호출을 지정하는 토큰을 포함한 모든 출력 토큰에 균일하게 적용되므로, 최종 보상이 답변 품질과 메모리 탐색 행동을 공동으로 최적화합니다.

```

       │
       ▼

│  ┌──────────────┐  ┌──────────────┐             │

          ▼
┌─────────────────────────────────────────────────┐
│          Multi-Granularity Memory Pyramid        │
│  ┌─────────────────────────────────────────┐   │
│  │ L4: User Profile (Markdown)             │   │
```

### 2.5 한계점

논문에서 명시적으로 언급된 한계:

1. **벤치마크 제한**: 기존 장기 메모리 벤치마크 기반 평가로 실제 세계의 모든 형태의 개인화와 개방형 사용자 상호작용을 포착하지 못할 수 있음

2. **프라이버시 및 망각 문제**: 사용자 메모리 시스템에서 발생하는 프라이버시 보호 및 망각(forgetting) 요구사항을 완전히 다루지 않음

3. **스케일링 연구 부재**: 모델 크기가 선호하는 메모리 접근 전략에 어떻게 영향을 미치는지에 대한 광범위한 스케일링 연구 부족

4. **증거 강도 보정 문제**: 케이스 스터디(Figure 6)에서 드러난 것처럼, 이벤트 수준 메모리를 안정적 선호도로 과잉 일반화하는 실패 사례 존재 → 탐색 정책이 증거 강도를 더 잘 보정해야 함

5. **도구 호출 예산 제한**: 추론 시 쿼리당 최대 4단계 도구 호출로 제한되어, 복잡한 크로스-세션 추론에서 충분한 증거를 수집하지 못할 가능성

## 3. 모델의 일반화 성능 향상 가능성

### 3.1 도메인 내 일반화 (In-Distribution Generalization)

NapMem은 LoCoMo와 PersonaMem-v2의 훈련 데이터로 학습하고, 동일 데이터셋의 **별도 사용자 분리(user-level split)된 테스트 세트**에서 평가하여 사용자 수준 누출을 방지합니다 (60/20/20 분할).

```
훈련 사용자 ≠ 테스트 사용자
→ 새로운 사용자의 메모리 패턴에 대한 일반화 능력 평가
```

**도메인 내 일반화의 강점 근거:**
- PersonaMem-v2: **47.97%** (최고 성능) — 암묵적 선호도와 페르소나 이해에서 강력한 일반화
- LoCoMo F1: **41.28%** — 사실 기반 장기 기억에서도 경쟁력 있는 성능

### 3.2 분포 외 일반화 (Out-of-Distribution Generalization)

**LongMemEval**은 RL 훈련에 전혀 사용되지 않은 **완전한 홀드아웃(held-out) 벤치마크**로, 진정한 분포 외 일반화를 측정합니다.

$$\text{LongMemEval}\ \notin\ \text{Training Data}$$

| 지표 | NapMem-9B w/ RL | 차선책(Mem0) |
|------|----------------|-------------|
| F1 | **57.41** | 53.86 |
| L-J | **80.33** | 78.00 |

**분포 외에서도 최고 성능을 달성**합니다. 이는 학습된 메모리 탐색 정책이 특정 데이터셋에 과적합되지 않고 일반적인 메모리 탐색 전략을 학습했음을 시사합니다.

### 3.3 비메모리 과제로의 일반화

메모리 특화 훈련이 다른 능력을 저해하지 않는지 검증:

**RL이 불필요한 메모리 호출을 억제하는 메커니즘:**

$$\text{Memory call (GPQA-D)}: \underbrace{34.51\%}_{\text{w/o RL}} \rightarrow \underbrace{6.90\%}_{\text{w/ RL}}$$

RL 훈련이 메모리가 필요하지 않은 상황에서 메모리 도구를 호출하지 않도록 정책을 보정합니다. 이는 과제 적응적(task-adaptive) 메모리 사용 패턴을 학습했음을 의미합니다.

**일반 추론 능력 보존:**
- GPQA-D: 53.03 (베이스) → **57.58** (w/ RL) — 오히려 향상
- BFCL-v3: 95.00 (베이스) → **95.00** (w/ RL) — 동등 유지
- V*Bench: 84.82 (베이스) → **91.10** (w/ RL) — 향상

### 3.4 모델 스케일에 따른 일반화 전략 차이

**도구 사용 행동 분석 (Figure 4, Table 5):**

| 모델 | 첫 번째 도구 (파일 읽기 비율) | 평균 도구 호출 | 증거 적중률 |
|------|---------------------------|--------------|------------|
| NapMem-9B | 3.96% | 3.97 | 20.66% |
| NapMem-9B w/ RL | 5.03% | **2.15** | **34.92%** |
| NapMem-122B | 44.38% | 2.50 | 26.20% |
| NapMem-397B | 46.60% | 2.69 | 26.42% |

- **대형 모델**: 더 자주 파일 읽기(상위 수준 메모리)로 시작 → 광범위한 주제 트랙/프로필 정보를 먼저 흡수
- **소형 모델**: 레코드 수준 도구 선호 → 더 짧고 목표 지향적인 증거부터 시작
- **RL 훈련**: 도구 호출 수 감소(3.97 → 2.15)와 동시에 증거 적중률 향상(20.66% → 34.92%) → 더 효율적이고 정확한 탐색

이는 **모델 크기와 훈련 모두 일반화 전략에 영향**을 주며, 소형 모델도 적절한 RL 훈련을 통해 대형 모델 수준의 성능을 달성할 수 있음을 보여줍니다.

### 3.5 일반화 성능 향상 가능성 종합 평가

```
일반화 성능 향상 가능성 지표:

1. 새로운 사용자 적응:  ✅ 사용자 수준 분리로 검증됨
2. 새로운 데이터셋:     ✅ LongMemEval OOD 최고 성능
3. 비메모리 과제:       ✅ 일반 능력 보존 및 일부 향상
4. 다양한 메모리 유형:  ✅ 사실/선호도/크로스-세션 모두 처리
5. 스케일 독립적 효과:  ✅ 9B 모델이 397B를 능가

잠재적 향상 방향:
- 증거 강도 보정 개선
- 더 긴 도구 호출 예산
- 프라이버시 보호 메모리 망각 메커니즘 통합
```

## 4. 향후 연구에 미치는 영향과 고려사항

### 4.1 향후 연구에 미치는 영향

#### 🔴 패러다임 전환 (Paradigm Shift)

NapMem은 메모리 시스템 연구의 핵심 패러다임을 다음과 같이 전환합니다:

```
Before (수동적):
[메모리 시스템] ─검색→ [컨텍스트] ─주입→ [LLM]

After (능동적):
[LLM 에이전트] ←→ [메모리 도구 인터페이스]
        ↑                    ↓
    [관측값 o_t]        [메모리 피라미드]
```

이 전환은 메모리 연구를 **검색 최적화 문제**에서 **의사결정 최적화 문제**로 재정의합니다.

#### 🟡 강화학습과 메모리의 결합

GRPO를 메모리 도구 사용 학습에 적용한 것은 향후 연구의 중요한 방향을 제시합니다:
- **메모리 관리 정책 학습**: 무엇을 저장하고, 언제 업데이트하고, 어떻게 검색할지를 RL로 동시 최적화
- **다중 에이전트 메모리 공유**: 여러 에이전트 간의 메모리 공유 정책 학습

#### 🟢 계층적 메모리 구조의 표준화

4계층 피라미드 구조는 향후 메모리 시스템 연구의 표준 설계 원칙으로 활용될 수 있습니다:
- **하위 계층**: 정밀 증거 검증
- **중간 계층**: 압축된 의미 정보
- **상위 계층**: 빠른 전역 컨텍스트 접근

#### 🔵 도구 사용 행동 분석 방법론

증거 적중률(evidence-hit ratio), 다중 수준 탐색 비율(multi-level navigation rate) 등의 새로운 평가 지표는 메모리 에이전트의 행동을 분석하는 표준 도구로 발전할 수 있습니다.

### 4.2 앞으로 연구 시 고려할 점

#### 🔑 핵심 고려사항

**[1] 증거 강도 보정 (Evidence Strength Calibration)**

논문의 실패 사례(Figure 6)에서 이벤트 메모리를 안정적 선호도로 과잉 일반화하는 문제가 드러났습니다. 향후 연구에서는:

$$\text{Confidence}(r) = f(\text{frequency}, \text{recency}, \text{record type}, \text{consistency})$$

과 같은 증거 강도 점수를 도입하여 이벤트 증거와 안정적 선호도 증거를 구분해야 합니다.

**[2] 프라이버시 보호 및 선택적 망각 (Privacy & Selective Forgetting)**

사용자 데이터를 장기 저장하는 시스템에서는 GDPR 등 규제 준수를 위한 선택적 메모리 삭제(Right to be Forgotten)와 차등 프라이버시(Differential Privacy) 적용이 필수적입니다:

$$\mathcal{M}_{\text{private}} = \text{DP-Sanitize}(\mathcal{M},\ \epsilon_{\text{privacy}})$$

**[3] 도구 호출 예산 최적화 (Tool Budget Optimization)**

현재 고정된 최대 4단계 도구 호출 예산을 적응적으로 결정하는 정책이 필요합니다:

$$T_{\max} = g(q, \mathcal{M}, \text{complexity estimate})$$

**[4] 다중 사용자 및 실시간 업데이트**

실제 배포 환경에서는 동시에 여러 사용자의 메모리를 관리하고 실시간으로 업데이트해야 합니다. 현재 증분 구축 방식의 확장성과 동시성 처리 능력 검증이 필요합니다.

**[5] 메모리 피라미드 자동 깊이 결정**

현재 4계층 구조는 수동으로 설계되었습니다. 사용자 특성과 과제 유형에 따라 최적의 계층 수와 구조를 자동으로 결정하는 메타 학습 접근법이 필요합니다.

**[6] 평가 벤치마크의 다양화**

현재 LoCoMo, LongMemEval, PersonaMem-v2는 특정 유형의 메모리 과제만 포함합니다. 향후 연구에서는:
- 다국어 메모리 과제
- 멀티모달 메모리 (이미지, 오디오 포함)
- 실시간 대화 스트리밍 환경
- 악의적 메모리 조작에 대한 견고성 평가

등이 필요합니다.

**[7] 메모리 구축 비용 최적화**

메모리 피라미드 구축에는 상당한 LLM 추론 비용이 소요됩니다. 저비용 모델로 메모리를 구축하고 고성능 모델로 탐색하는 비대칭 설계나, 캐싱 전략 등을 고려해야 합니다.

### 5.3 RL 기반 메모리 접근 연구 비교

| 연구 | RL 적용 대상 | 주요 차별점 vs NapMem |
|------|-------------|---------------------|
| **RMM** (Tan et al., 2025) | 검색 쿼리 최적화 | 검색 이후의 에이전트는 고정 인터페이스 사용 |
| **Search-R1** (Jin et al., 2025) | 검색 쿼리 생성 | 일반 웹 검색에 초점, 사용자 메모리 특화 아님 |
| **DeepRetrieval** (Jiang et al., 2025) | 검색 엔진 쿼리 | 검색 최적화, 메모리 탐색 정책 학습 아님 |
| **Mem-α** (Wang et al., 2025) | 메모리 구축 결정 | 구축 최적화, 검색 시 탐색 정책 학습 아님 |
| **AgeMem** (Yu et al., 2026) | 메모리 관리 통합 | 단순 플랫 메모리, 계층 탐색 없음 |
| **Memory-R1** (Yan et al., 2026) | 메모리 관리 및 활용 | 유사한 RL 접근, 다중 계층 피라미드 없음 |
| **NapMem** (Xu et al., 2026) | **쿼리 시 메모리 탐색** | **4계층 피라미드 + 능동적 탐색 정책 동시 학습** |

### 5.4 핵심 차별화 요소

NapMem이 기존 연구들과 차별화되는 핵심 요소는 다음 세 가지의 **동시 통합**입니다:

$$\text{NapMem} = \underbrace{\text{구조화된 저장}}_{\text{4계층 피라미드}} + \underbrace{\text{도구 인터페이스}}_{\text{5개 메모리 도구}} + \underbrace{\text{학습된 정책}}_{\text{GRPO RL}}$$

기존 연구들은 이 세 요소 중 하나 또는 둘만 다루었습니다:
- **Mem0, Zep, MemOS**: 구조화된 저장에 집중, 학습된 정책 없음
- **RMM, Search-R1**: 학습된 검색 정책에 집중, 구조화된 다중 계층 없음
- **AgeMem**: RL 정책 있으나 단순 플랫 메모리 구조
