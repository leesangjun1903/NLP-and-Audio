## 1. 핵심 주장과 주요 기여 요약

### 1.1 핵심 주장

LLM 에이전트가 수행하는 장기(long-horizon) 태스크(예: 웹 검색, 심층 리서치)에서, 누적된 컨텍스트는 **장기 컨텍스트 성능 저하(long-context degradation)**를 야기한다. 기존 연구들은 에이전트 자체가 컨텍스트를 관리하도록 하거나, 요약(summarization)과 같은 고정 전략을 사용하는데, 이는 에이전트 자체의 재훈련이 필요하고 다양한 에이전트에 범용 적용이 불가능하다.

본 논문은 **"에이전트를 전혀 수정하지 않고, 외부 관리자(external manager)를 훈련시켜 에이전트 친화적 컨텍스트 관리 전략을 학습할 수 있는가?"** 라는 질문에 답하며, 이를 **AdaCoM(Adaptive Context Management)** 으로 구현한다.

### 1.2 주요 기여

| 기여 | 내용 |
|---|---|
| **① AdaCoM 프레임워크 제안** | 에이전트를 동결(frozen)시킨 채 외부 LLM 관리자가 컨텍스트를 관리하는 아키텍처적 분리 구조 |
| **② 성능 향상** | 웹 검색, 심층 리서치 벤치마크에서 평균 39.0% 상대적 성능 향상 (ReAct 대비) |
| **③ Fidelity–Reliability Trade-off 발견** | 에이전트 능력에 따라 컨텍스트 보존 수준이 달라야 하는 원칙 규명 |
| **④ 크로스 에이전트 전이(Transfer) 가능성** | 능력 수준이 유사한 에이전트 간에 관리자를 재사용 가능함을 실험적으로 검증 |

## 2. 해결하고자 하는 문제, 제안 방법, 모델 구조, 성능, 한계

### 2.1 해결하고자 하는 문제

#### 장기 컨텍스트 성능 저하 (Long-Context Degradation)

ReAct 스타일의 에이전트는 다음과 같이 컨텍스트를 누적한다:

$$c_t^{\text{vanilla}} = (q, a_1, o_1, \ldots, a_t, o_t)$$

여기서 $q$는 태스크 쿼리, $a_i$는 에이전트의 추론 및 도구 호출, $o_i$는 환경 관측값이다. 이 **추가-전용(append-only) 워크플로우**는 컨텍스트가 누적됨에 따라:

- **관련 없는 콘텐츠**가 핵심 증거를 가림 (Shi et al., 2023)
- **위치 편향(positional bias)** 발생 (Liu et al., 2024)
- **중복 탐색** 및 **제약 망각(constraint forgetting)** 유발
- **조기 포기(premature abandonment)** 발생

기존 방법들의 한계:
- **SumAgent/MEM1**: 에이전트 자신이 요약을 수행 → 에이전트 재훈련 필요
- **MemAct**: 에이전트가 pruning 도구를 직접 호출 → 클로즈드 소스 에이전트에 부적합
- **고정 전략**: 에이전트 아키텍처/훈련 데이터의 다양성을 무시

---

### 2.2 제안하는 방법 (AdaCoM)

#### 2.2.1 AdaCoM 증강 워크플로우

관리자 정책 $\pi_\theta(\cdot \mid p)$를 도입하여, 에이전트 $\mathcal{A}$를 동결한 채 컨텍스트를 외부에서 제어한다.

**매 턴 $t$의 동작:**

1. 에이전트가 관리된 컨텍스트 $\tilde{c}_{t-1}$에서 행동 생성:

$$a_t \sim \mathcal{A}(\tilde{c}_{t-1})$$

2. 환경으로부터 관측값 $o_t$를 받아 사전 관리 컨텍스트 생성:

$$c_t = \text{Append}(\tilde{c}_{t-1}, a_t, o_t)$$

3. 관리자가 수정 행동 샘플링:

$$m_t \sim \pi_\theta(\cdot \mid p_t), \quad p_t = \mathcal{P}(c_t)$$

4. $m_t$를 $c_t$에 적용하여 다음 관리 컨텍스트 $\tilde{c}_t$ 생성

#### 2.2.2 유연한 행동 공간 (Action Space)

관리자 행동 $m_t$는 JSON 형식의 수정 연산 목록:

$$m_t = \left[\delta_t^{(1)}, \delta_t^{(2)}, \ldots, \delta_t^{(n_t)}\right]$$

각 연산 $\delta_t^{(j)}$는 4개 필드로 구성:
- $\text{ids}^{(j)}$: 대상 메시지 ID
- $\text{role}^{(j)} \in \{\text{SYSTEM}, \text{USER}, \text{ASSISTANT}\}$
- $\text{justification}^{(j)}$: 수정 근거 (에이전트에게는 비공개)
- $\text{new content}^{(j)}$: 대체 내용 (빈 문자열이면 삭제)

이 행동 공간은 **삭제(deletion)**, **요약(summarization)**, **병합(merging)**, **유지(no-change)** 등 다양한 연산을 자유롭게 지원한다.

---

### 2.3 훈련 방법 (RL 기반)

#### 2.3.1 MDP 공식화

관리자 학습을 **Markov Decision Process(MDP)**로 공식화:
- **상태**: 사전 관리 컨텍스트 $c_t$에서 생성된 프롬프트 $p_t$
- **행동**: 수정 행동 $m_t$
- **보상**: 에이전트의 최종 답변에 대한 태스크 보상

롤아웃: $\tau = ((p_1, m_1), \ldots, (p_T, m_T))$

#### 2.3.2 훈련 절차

1. **SFT 워밍업**: GPT-5 및 Claude Opus 4.6으로 생성한 편집 궤적으로 Qwen3-4B-Instruct 초기화
2. **GRPO 기반 RL 최적화** (Shao et al., 2024)

#### 2.3.3 2단계 어드밴티지 추정 (Two-Level Advantage Estimation)

**태스크 수준 어드밴티지** (결과 보상 정규화):

$$A_i^R = \frac{R_i - \mu_R}{\sigma_R + \varepsilon}$$

**스텝 수준 어드밴티지** (과정 보상 정규화):

$$A_{i,t}^Q = \frac{Q_{i,t} - \mu_Q}{\sigma_Q + \varepsilon}$$

여기서 $\mu_R, \sigma_R$은 그룹 $\{R_j\}_{j=1}^G$의 평균/표준편차, $\mu_Q, \sigma_Q$는 모든 스텝에서 계산.

**두 수준 결합**:

$$A_{i,t} = A_i^R + \alpha A_{i,t}^Q$$

**재정규화**:

$$\hat{A}_{i,t} = \frac{A_{i,t} - \mu_A}{\sigma_A + \varepsilon}$$

#### 2.3.4 과정 보상 (Process Rewards)

| 보상 유형 | 값 | 조건 |
|---|---|---|
| 토큰 패널티 | $-0.8$ | 컨텍스트가 길이 제한 초과 |
| 중복 행동 패널티 | $-0.4$ | 연속 동일 도구 호출 |
| 포맷 패널티 | $-0.5$ | JSON 파싱 실패 등 |
| gold-doc-found | $+0.6$ | 검색 결과에 정답 문서 포함 |
| key-doc-found | $+0.3$ | 검색 결과에 핵심 문서 포함 |

#### 2.3.5 정책 최적화 (PPO-style)

$$\mathcal{J}(\theta) = \mathbb{E}\left[\frac{1}{Z}\sum_{i,t,u}\min\left(r_{i,t,u}(\theta)\hat{A}_{i,t},\ \bar{r}_{i,t,u}(\theta)\hat{A}_{i,t}\right)\right]$$

여기서:
- $r_{i,t,u}(\theta)$: 토큰 수준 중요도 비율 ($\pi_\theta$ vs $\pi_{\theta_{\text{old}}}$)
- $\bar{r}\_{i,t,u}(\theta) = \text{clip}(r_{i,t,u}(\theta), 1-\epsilon, 1+\epsilon)$
- $Z = \sum_{i=1}^G \sum_{t=1}^{T_i} |m_{i,t}|$: 총 관리자 발생 토큰 수

### 2.4 모델 구조

```
┌─────────────────────────────────────────────────────┐
│                   환경 (Environment)                  │
│                검색 엔진 / Wikipedia MCP              │
└──────────────────────┬──────────────────────────────┘
                       │ 관측값 o_t
                       ▼
┌─────────────────────────────────────────────────────┐
│          사전 관리 컨텍스트 c_t                        │
│    = Append(c̃_{t-1}, a_t, o_t)                      │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│         AdaCoM 관리자 (Qwen3-4B-Instruct)            │
│         정책: π_θ(· | p_t)                           │
│         훈련: SFT + GRPO (RL)                        │
│         입력: 32K 토큰 컨텍스트 윈도우                  │
│         출력: JSON 수정 행동 m_t (최대 4K 토큰)         │
└──────────────────────┬──────────────────────────────┘
                       │ 수정된 컨텍스트 c̃_t
                       ▼
┌─────────────────────────────────────────────────────┐
│         동결 에이전트 ❄ (Frozen Agent)                │
│    (Qwen3-Max / DeepSeek-V3 / Kimi-K2 / GLM-4.5)   │
│         행동: a_{t+1} ~ A(c̃_t)                      │
└─────────────────────────────────────────────────────┘
```

**핵심 구조적 특징:**
- 관리자(Qwen3-4B-Instruct)와 에이전트(클로즈드 소스 가능)의 **완전한 아키텍처 분리**
- 관리자는 에이전트를 블랙박스로 취급 → 클로즈드 소스 에이전트 지원
- 태스크 피드백(보상)이 **관리자만 업데이트**

## 3. 모델의 일반화 성능 향상 가능성

### 3.1 Fidelity–Reliability Trade-off: 일반화의 이론적 기반

논문이 발견한 핵심 원칙인 **Fidelity–Reliability Trade-off**는 일반화 성능의 구조적 토대가 된다:

$$\text{에이전트 능력} \uparrow \Rightarrow \text{컨텍스트 보존 길이} \uparrow \quad (\text{Tiered Management})$$
$$\text{에이전트 능력} \downarrow \Rightarrow \text{공격적 압축} \uparrow \quad (\text{Eager Distillation})$$

실험에서 관찰된 에이전트별 평균 사후 관리 컨텍스트 길이:

| 에이전트 | ReAct 성능 순위 | 평균 컨텍스트 길이 |
|---|---|---|
| DeepSeek-V3 | 낮음 | ~1.9K 토큰 |
| Kimi-K2 | 중하 | ~3.4K 토큰 |
| Qwen3-Max | 중상 | ~5.2K 토큰 |
| GLM-4.5-Air | 높음 | ~7.0K 토큰 |

이 트레이드오프가 일반화에 주는 함의: **에이전트의 "효과적 컨텍스트 길이(effective context length)"** 라는 보편적 개념이 존재하며, 이것이 관리 전략의 핵심 파라미터가 됨을 시사한다.

### 3.2 능력 기반 전이(Capability-Based Transfer)와 일반화

**크로스 에이전트 전이 결과**는 일반화의 직접적 증거를 제공한다:

```
훈련 소스 에이전트의 능력 ≈ 대상 에이전트의 능력
→ 전이 성능 ↑
```

- **고성능 대상** (GLM, Qwen, Gemini, GPT-OSS-20B): 고성능 소스(GLM, Qwen) 관리자 선호
- **저성능 대상** (Kimi, DeepSeek, Seed, GPT-4o-mini): 저성능 소스(DeepSeek, Kimi) 관리자 선호

이는 능력 수준이라는 **단일 프록시 지표**를 통해 관리자 선택을 일반화할 수 있음을 의미한다.

### 3.3 일반화 성능을 제한하는 요인들

논문이 확인한 전이 실패 사례들은 일반화의 한계와 향후 개선 방향을 보여준다:

#### (a) 작업 기억 스타일 불일치
- **DeepSeek**: 간결한 핵심 발견+문서 ID 선호
- **Kimi**: 상세한 검색 이력 선호
- → 능력이 유사해도 스타일 불일치 시 전이 실패

#### (b) 공격적 압축 민감도
- **GLM**: 다른 에이전트용 관리자들이 너무 공격적으로 압축 → 세부 정보 손실
- Qwen3-4B 관리자의 낙손 없는 패러프레이징 한계

#### (c) 출력 형식 호환성
- **Gemini**: AdaCoM(Qwen)의 보고서 형식 기억을 받으면 검색 대신 보고서 생성 모드로 전환
- 관리자의 수정 스타일이 대상 에이전트의 추론 행동과 상호작용

### 3.4 일반화 성능 향상 가능성 (논문 기반 추론)

논문이 제시한 방향과 실험 결과로부터 도출할 수 있는 일반화 향상 가능성:

| 방향 | 근거 |
|---|---|
| **더 큰 관리자 모델 사용** | 4B 관리자의 용량 한계 명시 → 더 큰 모델로 무손실 압축 가능성 ↑ |
| **단계적 관리(Tiered Management) 원칙화** | 강한 에이전트에는 간헐적 개입, 약한 에이전트에는 매 스텝 압축 |
| **스타일 프로파일링** | 에이전트별 추론 스타일을 사전 분류하여 관리자 선택 정교화 |
| **능력 프록시 다양화** | 현재 vanilla ReAct 성능 단일 지표 → 다차원 능력 측정으로 확장 |

## 4. 앞으로의 연구에 미치는 영향과 연구 시 고려할 점

### 4.1 앞으로의 연구에 미치는 영향

#### (1) 에이전트 시스템 설계 패러다임 전환
AdaCoM은 **"에이전트를 수정하지 않는 외부 모듈로 에이전트의 한계를 보완"**한다는 새로운 패러다임을 제시한다. 이는:
- 클로즈드 소스 LLM을 에이전트로 활용하는 실용적 시스템 설계에 직접 적용 가능
- 플러그인형 모듈 구조(에이전트-관리자 분리)를 표준 인터페이스로 발전시킬 수 있음
- "에이전트 오케스트레이션" 연구(multi-agent systems)에서 컨텍스트 관리 역할의 독립 모듈화를 촉진

#### (2) RL 기반 LLM 에이전트 훈련 연구
- **2단계 어드밴티지 추정**의 도입은 희소 보상(sparse reward) 문제를 완화하는 실용적 방법으로, 유사한 장기 태스크 RL 훈련에 적용 가능
- 과정 보상 설계(process reward design)의 규칙 기반 접근법은 LLM 심판(judge) 없이도 중간 신호를 제공하는 방법론적 기여

#### (3) Fidelity–Reliability Trade-off의 이론화
이 발견은 LLM의 **효과적 컨텍스트 길이(effective context length)** 개념을 실증적으로 지지하며:
- 장기 컨텍스트 LLM 평가 방법론 연구에 영향
- 에이전트 훈련 시 컨텍스트 관리 능력을 목표로 포함하도록 유도

#### (4) 전이 학습과 재사용 가능한 에이전트 구성요소
능력 기반 전이 패턴의 발견은:
- "에이전트 라이브러리"에서 관리자를 재사용하는 실용적 배포 전략 수립에 기여
- 적은 수의 대표 에이전트로 관리자를 훈련하고 광범위하게 적용하는 **효율적 에이전트 생태계** 구축 가능성

---

### 4.2 앞으로 연구 시 고려할 점

#### (1) 관리자 용량과 대상 에이전트 능력의 스케일링
- 현재 4B 관리자는 강력한 에이전트(예: GPT-5급)에 대해 한계를 보임
- **연구 과제**: 관리자 크기와 전이 성능 간의 스케일링 법칙(scaling law) 규명
- 관리자가 대상 에이전트보다 훨씬 약한 경우의 정보 손실 하한 분석 필요

#### (2) 더 다양한 장기 태스크로의 일반화 검증
- 현재 평가: 지식 집약적 검색(웹 검색, 위키피디아 리서치)에 한정
- **고려할 태스크**: 코드 에이전트(SWE-bench 등), 구현 에이전트, 수학적 추론, 장문 작성
- 각 태스크 유형에서 "유용한 컨텍스트"의 구조적 차이 분석 필요

#### (3) KV 캐시 효율과 컨텍스트 관리의 공동 최적화
- AdaCoM의 컨텍스트 수정은 KV 캐시 재사용을 방해함 (Kwon et al., 2023)
- **연구 방향**: 캐시 무효화를 최소화하는 "캐시 친화적 수정 전략" 설계
  - 컨텍스트의 앞부분(초기 질문, 중요 제약)은 고정 보존
  - 뒷부분만 선택적으로 압축하는 구조 탐색

#### (4) 관리자 스타일과 에이전트 추론 스타일의 호환성 자동 평가
- 현재: 전이 실패 사례를 사후(post-hoc) 궤적 분석으로만 확인
- **연구 과제**: 관리자-에이전트 스타일 호환성을 **훈련 전 사전 평가**하는 자동화 메트릭 개발
  - 예: 관리자 출력 스타일과 에이전트 입력 선호도 간의 분포 거리 측정

#### (5) 과정 보상의 자동 설계
- 현재 과정 보상은 수작업으로 설계된 규칙 기반
- **연구 방향**: 태스크별 과정 보상을 자동으로 발견하는 메타 학습 접근법

#### (6) 능력 측정 지표의 다양화
- 현재 "에이전트 능력" 프록시: vanilla ReAct 성능(단일 스칼라)
- **더 정교한 측정**: 추론 깊이, 도구 활용 효율성, 오류 복구 능력 등 다차원 능력 프로파일
- 이를 기반으로 한 더 정확한 관리자-에이전트 매칭 전략

#### (7) 온라인/적응적 관리자 업데이트
- 현재: 관리자를 오프라인으로 훈련 후 고정 배포
- **연구 방향**: 배포 중 새로운 에이전트-태스크 조합을 접할 때 관리자를 소량의 상호작용으로 빠르게 적응시키는 **온라인 적응(online adaptation)** 연구

### 5.1 AdaCoM의 차별적 위치

```
에이전트 내부 방법론        외부 방법론
(에이전트 훈련 필요)      (에이전트 독립적)
        │                        │
MEM1, IterResearch         AdaCoM ← 유일하게
MemAct, Memory-R1          유연한 행동공간 +
                           RL 훈련 조합
        │                        │
   고정 전략              유연 전략
 (요약/pruning)          (삭제/요약/병합/유지)
```

### 5.2 장기 메모리 연구와의 차별성

Mem0, A-Mem, G-Memory 등은 **세션 간(cross-session) 장기 메모리**를 다루는 반면, AdaCoM은 **단일 장기 태스크 내 작업 메모리(working memory) 관리**에 집중한다. 이 두 방향은 상호 보완적이며, 향후 통합 연구가 가능하다.
