## 1. 핵심 주장 및 주요 기여 요약

### 핵심 주장

AI 에이전트의 성능은 런타임 **하네스(harness)** — 즉 모델이 관찰하고, 추론하고, 행동하는 방식을 매개하는 프롬프트, 도구, 메모리, 제어 흐름 — 에 결정적으로 의존한다. 그러나 오늘날의 하네스는 대부분 수동으로 제작(hand-crafted)되고 정적(static)이어서, 새로운 모델이나 태스크마다 맞춤형 스캐폴딩을 다시 작성해야 하며, 실행 중 생성된 풍부한 트레이스(traces)는 체계적인 개선으로 환류되지 않는다.

이 논문의 핵심 주장은 **에이전트의 발전이 반드시 모델 스케일링에서만 올 필요는 없으며**, 실행 피드백으로부터 런타임 인터페이스를 구성(composing)하고 진화(evolving)시키는 것이 실행 가능하고 상보적인 레버라는 것이다.

### 주요 기여

논문은 **HarnessX**를 제안하며, 이는 다음 세 가지 핵심 메커니즘으로 구성된다:
1. **타입화된 하네스 프리미티브(typed harness primitives)**를 **치환 대수(substitution algebra)**를 통해 조립
2. **AEGIS**: 기호적 적응(symbolic adaptation)과 강화 학습 간의 운영적 미러(operational mirror)에 기반한 트레이스 구동 다중 에이전트 진화 엔진
3. 궤적(trajectories)을 하네스 업데이트와 모델 학습 신호 모두로 전환하는 **하네스-모델 루프 폐쇄**

## 2. 해결하고자 하는 문제, 제안 방법, 모델 구조, 성능, 한계

### 2.1 해결하고자 하는 문제

기존 에이전트 시스템은 두 가지 근본적인 문제를 가진다:
- 첫째, 하네스를 타입화된 요소로 구성된 대체 가능한 엔티티로 다루는 레이어가 없어, 태스크별 하네스 구축 시 항상 재작성이 필요하다.
- 둘째, 인-루프(in-loop) 개선 메커니즘이 존재하지 않아, 하네스는 릴리즈 사이 인간의 반복을 통해서만 진화한다.

Claude Code의 Dynamic Workflows처럼 런타임에 태스크별 하네스 스크립트를 생성하는 시도도 있었으나, 이는 단일 세션 내에서만 동작하며 영속적(persistent) 트레이스 기반 최적화, 교차 세션(cross-session) 진화, 하네스-모델 공동 훈련(co-training)이 결여되어 있다.

### 2.2 제안하는 방법

#### (1) 치환 대수 기반 하네스 조립

HarnessX는 하네스를 타입화된 프리미티브와 치환 대수를 사용한 **조립 가능한 1급 객체(composable first-class object)**로 다룬다.

하네스 $\mathcal{H}$는 타입화된 프리미티브들의 집합으로 정의되며, 치환 대수의 합성 연산은 다음과 같이 표현할 수 있다:

$$\mathcal{H} = \bigoplus_{i} p_i, \quad p_i \in \mathcal{P}_{\tau_i}$$

여기서 $p_i$는 타입 $\tau_i$ (예: 프롬프트, 도구, 메모리, 제어 흐름)에 속하는 프리미티브이고, $\bigoplus$는 치환 대수 상의 합성 연산이다.

두 하네스 $\mathcal{H}_1$, $\mathcal{H}_2$의 합성 시:

$$\mathcal{H}_{1} \circ \mathcal{H}_{2} = \text{subst}(\mathcal{H}_{1}, \mathcal{H}_{2})$$

즉, 특정 타입 슬롯에 대한 프리미티브를 교체함으로써 변형 하네스(variant harness)를 격리·테스트할 수 있다.

#### (2) AEGIS: 트레이스 구동 다중 에이전트 진화 엔진

AEGIS는 **4단계 파이프라인**으로 구성된다: Digester, Planner, Evolver, Critic. 각각 트레이스 압축, 적응 계획, 후보 생성, 변경 평가의 역할을 담당한다.

AEGIS의 진화 목표는 다음과 같이 정식화할 수 있다:

$$\mathcal{H}^{*} = \arg\max_{\mathcal{H}} \mathbb{E}_{\tau \sim \pi_{\mathcal{H}}} \left[ R(\tau) \right]$$

여기서 $\tau$는 실행 트레이스, $\pi_{\mathcal{H}}$는 하네스 $\mathcal{H}$ 하에서의 에이전트 정책, $R(\tau)$는 태스크 보상이다.

각 AEGIS 라운드 $t$에서의 업데이트는:

$$\mathcal{H}^{(t+1)} = \text{Evolver}\left(\text{Planner}\left(\text{Digester}(\mathcal{T}^{(t)})\right)\right)$$

단, Critic이 검증을 통과한 경우에만 $\mathcal{H}^{(t+1)}$을 채택한다.

#### (3) 하네스-모델 공동 진화 (Harness-Model Co-Evolution)

하네스 적응과 모델 훈련 사이의 루프를 닫는 **하네스-모델 공동 진화**를 수행한다. 하네스 적응 중 생성된 트레이스가 모델 훈련을 위한 강화 학습 신호로 활용되어, 모델 개선이 이후 하네스 진화로 다시 피드백된다.

이를 수식으로 표현하면:

$$\theta^{(t+1)} = \text{RL-Train}\left(\theta^{(t)},\ \mathcal{B}^{(t)}\right), \quad \mathcal{B}^{(t)} = \bigcup_{k \leq t} \mathcal{T}^{(k)}$$

여기서 $\theta$는 기반 모델 파라미터, $\mathcal{B}^{(t)}$는 라운드 $t$까지의 공유 재생 버퍼(replay buffer)이다.

HarnessX는 타입화된 합성에 의한 변형 격리, 병리 감지를 위한 구조적 관찰 가능성(structured observability), 하네스 진화와 모델 훈련 간 루프를 닫는 공유 재생 버퍼를 갖춘 **다중 라운드, 트레이스 구동 학습 문제**로 하네스 적응을 다룬다.

### 2.3 모델 구조

HarnessX는 **조합 가능한 프리미티브, 트레이스 구동 진화, 피드백 루프**를 통해 적응적이고 진화 가능한 AI 에이전트 런타임 인터페이스를 활성화한다.

구조를 정리하면:

| 계층 | 구성 요소 | 역할 |
|------|-----------|------|
| **조합 계층** | Typed Harness Primitives + Substitution Algebra | 하네스를 타입별로 조립·교체 |
| **진화 계층** | AEGIS (Digester → Planner → Evolver → Critic) | 트레이스 기반 하네스 적응 |
| **공동 진화 계층** | Harness-Model Loop (Replay Buffer) | 하네스 업데이트 ↔ 모델 RL 훈련 신호 |

### 2.4 성능

다섯 개의 벤치마크(ALFWorld, GAIA, WebShop, $\tau^3$-Bench, SWE-bench Verified)에 걸쳐 HarnessX는 **평균 +14.5%**(최대 **+44.0%**)의 성능 향상을 달성했으며, 기준선이 낮을수록 향상폭이 컸다.

### 2.5 한계

AEGIS 루프가 타입화된 프로세서를 통해 하네스 프리미티브를 편집하지만, **+14.5% 평균 및 +44.0% 최대 성능 향상에는 중요한 주의점이 있다: 모든 성능 향상이 진화에 사용된 동일한 태스크 세트에서 측정되었다는 점이다.**

검증 게이트(validation gate)가 없을 경우, "루프는 단순히 자동화된 과적합(overfitting) 방법"이 될 위험이 있다.

추가 한계:
- 전체 코드베이스는 미래 릴리즈에서 오픈소스로 공개 예정이며, 현재는 공개되지 않았다.

## 3. 일반화 성능 향상 가능성

### 3.1 일반화를 지원하는 요인

타입화된 합성에 의한 **변형 격리(variant isolation)**는 개별 프리미티브의 효과를 격리하여 과적합 없이 구조적으로 테스트할 수 있게 하며, 구조적 관찰 가능성(structured observability)은 특정 도메인에 국한되지 않는 병리 감지를 가능하게 한다.

HarnessX는 하네스 적응을 **다중 라운드, 트레이스 구동 학습 문제**로 다루기 때문에, 단일 태스크/모델에 특화된 해결책이 아닌 범용적 적응 프레임워크로 설계되었다.

다중 도메인 벤치마크(ALFWorld: 가정용 로봇, GAIA: 범용 AI, WebShop: 전자상거래, $\tau^3$-Bench, SWE-bench: 소프트웨어 엔지니어링)에 걸친 일관된 성능 향상은 도메인 전반에 걸친 일반화 가능성을 시사한다.

비교 연구인 HarnessFix([arXiv:2406.XXX])의 결과도 이를 지지한다:
트레이스 기반 하네스 수리에서의 성능 향상이 **네 가지 서로 다른 도메인**(저장소 수준 패칭, 커맨드라인 워크플로우, 오픈엔드 리서치, 상태 기반 애플리케이션 자동화)에서 나타났다는 사실은 트레이스 기반 하네스 수정이 단일 벤치마크 계열을 넘어 일반화됨을 나타낸다.

### 3.2 일반화의 한계

AEGIS의 성능 향상이 **진화에 사용된 동일한 태스크 세트**에서 측정되었기 때문에, 완전한 홀드아웃(held-out) 분할에서의 일반화 성능은 명확히 검증되지 않았다.

### 3.3 하네스-모델 공동 진화와 일반화

하네스 적응 중 생성된 트레이스가 모델 훈련을 위한 강화 학습 신호로 사용되어 **모델 개선이 이후 하네스 진화로 피드백**되는 공동 진화 메커니즘은, 다양한 태스크 분포로의 전이 가능성을 높이는 잠재적 일반화 촉진 요인이다.

또한 관련 연구에서:
모델의 기본 역량과 하네스 자기 진화 역량 간의 관계를 분석한 결과, **하네스 업데이트 역량은 기본 역량에 거의 무관하다**: 서로 다른 역량 수준의 모델들이 놀랍도록 유사한 성능 향상을 가져오는 하네스 업데이트를 생성한다. 이는 HarnessX 방식이 다양한 규모의 모델에서도 일반화될 수 있음을 시사한다.

## 4. 향후 연구에 미치는 영향 및 고려 사항

### 4.1 향후 연구에 미치는 영향

#### (1) 패러다임 전환: 모델 스케일링 → 하네스 진화
이 논문의 결과는 에이전트 발전이 반드시 모델 스케일링에서만 올 필요가 없으며, **실행 피드백으로부터 런타임 인터페이스를 구성하고 진화시키는 것이 실행 가능하고 상보적인 레버**임을 보여준다. 이는 고비용의 대규모 모델 훈련 없이도 에이전트 성능을 향상시킬 수 있는 새로운 연구 방향을 제시한다.

#### (2) 하네스를 1급 객체로 다루는 연구 흐름 촉발
만약 스캐폴딩이 자체 트레이스로부터 스스로를 구성하고 개선할 수 있다면, 에이전트 엔지니어링을 조용히 지배하던 모델별 재작성 비용이 사라지기 시작한다. 이는 하네스를 **1급 프로그래밍 가능한 아티팩트로 다루는 가장 깔끔한 버전**이다.

#### (3) 자기 진화 에이전트 연구의 체계화
자기 진화 에이전트에 관한 연구는 기반 모델을 재훈련하지 않고 에이전트 시스템을 개선하는 방법을 조사해왔으며, 초기 연구들은 가장 편집하기 쉬운 측면인 프롬프트에만 집중했다(APE, OPRO, EvoPrompt, Promptbreeder 등). HarnessX는 이를 프롬프트를 넘어 도구, 메모리, 제어 흐름 전체로 확장한다.

### 4.2 향후 연구 시 고려 사항

| 고려 사항 | 상세 설명 |
|-----------|-----------|
| **홀드아웃 일반화 검증** | 진화에 사용된 태스크와 분리된 별도의 홀드아웃 분할에서의 성능 검증 필수 |
| **과적합 방지 게이트 설계** | Self-Harness는 홀드인/홀드아웃 분할 모두에서 회귀하지 않고 적어도 하나를 개선할 때만 후보를 채택하는 방식을 채택한다. 유사한 안전 게이트 연구가 필요하다 |
| **비용-효율 트레이드오프** | AEGIS 4단계 파이프라인 및 다중 에이전트 진화의 계산 비용 vs. 성능 향상의 균형 분석 |
| **모델-하네스 결합도** | 특정 기반 모델에 과적합된 하네스가 다른 모델로 전이될 때의 성능 저하 연구 |
| **보안 및 안전성** | 하네스가 자기 진화할 경우 예기치 않은 행동 발생 가능성에 대한 안전성 검증 메커니즘 |
| **하네스 설명 가능성** | 어떤 프리미티브 변화가 성능 향상을 이끌었는지 귀인 분석(attribution analysis) 연구 |

| **Harness Updating Is Not Harness Benefit** | 2026 | 하네스 자기 진화 분석 | 하네스 업데이트 역량과 하네스 혜택 역량을 분리 분석 | 분석 연구, 새 시스템 제안 없음 |
| **AHE (Agentic Harness Engineering)** | 2026 | 관찰 가능성 기반 코딩 에이전트 하네스 진화 | 다섯 개의 성능 향상이 모두 양수로 단일 제공자 관용어나 단일 추론 깊이에 특정되지 않음 | 코딩 도메인에 특화 |
| **HarnessFix** | 2026 | 트레이스 기반 하네스 결함 진단·수리 | SWE-Bench Verified에서 45→57 해결(26.7% 상대적 향상) | 수리 초점, 공동 진화 없음 |
| **HarnessX** | 2026 | 조합·적응·진화 하네스 파운드리 | 치환 대수 + AEGIS + 하네스-모델 공동 진화 | **하네스 전체 구성 요소 + 모델 공동 훈련 통합** |
