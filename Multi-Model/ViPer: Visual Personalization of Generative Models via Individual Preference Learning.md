## 1. 핵심 주장 및 주요 기여 요약

### 🔑 핵심 주장

동일한 프롬프트로 생성된 이미지라도 서로 다른 사용자는 서로 다른 이미지를 선호한다. 이로 인해 개인의 시각적 선호도에 맞는 이미지를 생성하는 **개인화 이미지 생성(Personalized Image Generation)** 이 필요하다. 그러나 현재 생성 모델들은 광범위한 대중에게 어필하는 출력을 생성하도록 조정되어 있으며, 개인 사용자에게 맞는 이미지를 생성하기 위해서는 사용자가 반복적으로 수동 프롬프트 엔지니어링을 수행해야 하는데, 이는 비효율적이고 바람직하지 않다.

### ✅ 주요 기여

| 기여 항목 | 설명 |
|---|---|
| **일회성 선호 수집** | 소수의 이미지에 대한 자유형식 코멘트로 사용자 선호 캡처 |
| **VPE 모듈** | IDEFICS2-8B 기반의 비전-언어 모델로 구조화된 선호 속성 추출 |
| **추가 파인튜닝 불필요** | Stable Diffusion 모델을 재학습 없이 조건부 생성 가능 |
| **개인화 강도 조절** | 슬라이더 도구를 통한 개인화 정도 조절 기능 제공 |
| **Proxy 평가 지표** | 개인화 생성물 평가를 위한 대리 척도 학습 |

ViPer는 동일한 프롬프트에 대해 서로 다른 사용자의 시각적 선호도에 맞게 생성 모델의 출력을 개인화하는 방법으로, 사용자의 일반적인 선호도를 **일회성(one-time)**으로 캡처하고, 상세한 프롬프트를 엔지니어링할 필요 없이 생성 모델을 조건화하여 수행된다.

## 2. 해결하고자 하는 문제, 제안 방법, 모델 구조, 성능 및 한계

### 2-1. 해결하고자 하는 문제

개인화 이미지 생성은 개인의 시각적 선호도에 맞는 이미지를 만드는 것을 포함한다. 그러나 현재의 생성 모델들은 광범위한 대중에게 어필하는 출력을 생성하도록 조정되어 있으며, 개인 사용자의 시각적 선호도로의 개인화는 반복적이고 수동적인 프롬프트 엔지니어링에 의존하는데, 이는 시간적으로도 비효율적이고 확장 가능하지 않다.

구체적으로 해결하려는 문제는 다음 세 가지이다:

1. **비효율성**: 매 이미지 생성 시 프롬프트를 수동으로 조정해야 함
2. **비확장성**: 사용자가 늘어날수록 반복 작업이 폭발적으로 증가
3. **비구조화된 선호**: 사용자의 선호는 암묵적이고 언어화하기 어려움

---

### 2-2. 제안하는 방법 (수식 포함)

ViPer의 파이프라인은 크게 **3단계**로 구성된다.

#### **Step 1: 사용자 코멘트 수집 (Free-form Comment Collection)**

먼저 사용자가 다양한 스타일과 개념의 소규모 이미지 선택에 대해 각 이미지를 좋아하거나 싫어하는 이유를 설명하는 코멘트를 제공하는 일회성 과정을 통해 사용자의 일반적인 선호도를 캡처한다. 이러한 코멘트를 바탕으로, 대규모 비전-언어 모델을 사용하여 사용자가 좋아하고 싫어하는 구조화된 시각적 속성, 즉 그들의 시각적 선호도를 추론한다.

코멘트는 특정 구조를 가질 필요가 없으며, 사용자는 원하는 만큼 많거나 적게 작성할 수 있다. 더 자세하고 표현적인 코멘트는 더 나은 개인화로 이어진다.

#### **Step 2: Visual Preference Extractor (VPE)**

자유 형식 코멘트를 처리하기 위해 비전-언어 모델을 사용하여 구조화된 선호도, 즉 **Visual Preference Extractor(VPE)**를 추출한다. VPE를 구축하기 위해 시뮬레이션된 에이전트와 그들의 코멘트 및 선호도 데이터셋에서 **IDEFICS2-8B**를 파인튜닝한다.

VPE의 역할을 수식으로 표현하면:

$$
\text{VPE}(\mathcal{I}, \mathcal{C}) \rightarrow \mathbf{VP}^+ , \mathbf{VP}^-
$$

여기서:
- $\mathcal{I} = \{i_1, i_2, \ldots, i_n\}$: 사용자가 코멘트한 이미지 집합
- $\mathcal{C} = \{c_1, c_2, \ldots, c_n\}$: 각 이미지에 대한 자유형식 코멘트
- $\mathbf{VP}^+$: 사용자가 **좋아하는** 시각적 속성 집합 (예: surrealism, rough brushstrokes)
- $\mathbf{VP}^-$: 사용자가 **싫어하는** 시각적 속성 집합 (예: monochromatic, overly realistic)

이 코멘트들은 시각적 속성을 포함하는 구조화된 표현, 즉 이미지에서 좋아하거나 싫어하는 특징들의 집합(예: surrealism, rough brushstrokes, monochromatic 등)으로 변환된다.

#### **Step 3: 텍스트-이미지 모델 조건화**

사용자의 자유 형식 코멘트를 구조화된 시각적 선호도로 변환하는 것은 개인의 선호도에 대한 간결한 표현을 제공하는 일회성 과정이다. 그런 다음 사용자의 시각적 선호도는 인코딩되어 **프롬프트 임베딩에 추가**된다.

이를 수식으로 표현하면, 최종 조건화된 프롬프트 임베딩 $\mathbf{e}_{cond}$는:

$$
\mathbf{e}_{cond} = \mathbf{e}_{prompt} \oplus \alpha \cdot \mathbf{e}_{VP^+} \ominus \beta \cdot \mathbf{e}_{VP^-}
$$

여기서:
- $\mathbf{e}_{prompt}$: 원본 텍스트 프롬프트 임베딩
- $\mathbf{e}_{VP^+}$: 긍정 시각 속성의 CLIP 텍스트 임베딩
- $\mathbf{e}_{VP^-}$: 부정 시각 속성의 CLIP 텍스트 임베딩
- $\alpha, \beta$: 개인화 강도 조절 파라미터

그리고 개인화된 이미지 생성은 **Classifier-Free Guidance (CFG)**를 확장하여:

$$
\hat{\epsilon}_\theta(\mathbf{x}_t, \mathbf{e}_{cond}) = \epsilon_\theta(\mathbf{x}_t, \emptyset) + w \cdot \left[\epsilon_\theta(\mathbf{x}_t, \mathbf{e}_{cond}) - \epsilon_\theta(\mathbf{x}_t, \emptyset)\right]
$$

여기서 $w$는 guidance scale, $\epsilon_\theta$는 노이즈 예측 네트워크를 의미한다.

이 접근 방식은 사용자가 단순한 슬라이더 도구로 개인화 수준을 조절할 수 있는 유연성을 제공하며, 개인화 정도가 0에서 1로 증가함에 따라 결과가 사용자 선호도에 맞게 더욱 맞춤화된다.

#### **Step 4: Proxy 평가 지표 학습**

Proxy 지표는 개인의 좋아하고 싫어하는 이미지 집합과 쿼리 이미지가 주어졌을 때, 해당 사용자가 쿼리 이미지를 얼마나 좋아할지를 나타내는 **선호 점수를 예측**할 수 있는 모델이다.

이 프록시 측도를 얻기 위해 에이전트의 데이터와 **Pick-a-Pic 데이터셋** 두 가지 소스의 훈련 데이터를 결합한다. 후자는 생성된 이미지 집합에 대한 인간 사용자의 이진 선호도를 포함한다.

모델은 **교차 엔트로피 손실(cross-entropy loss)**로 학습된다.

프록시 지표의 손실 함수:

$$
\mathcal{L}_{proxy} = -\frac{1}{N}\sum_{i=1}^{N}\left[y_i \log \hat{p}_i + (1-y_i)\log(1-\hat{p}_i)\right]
$$

여기서 $y_i \in \{0, 1\}$은 사용자의 선호 여부, $\hat{p}_i$는 모델이 예측한 선호 확률이다.

### 2-3. 모델 구조

```
┌─────────────────────────────────────────────────────┐
│                  ViPer Pipeline                     │
│                                                     │
│  [이미지 집합 I] + [자유형식 코멘트 C]              │
│           │                                         │
│           ▼                                         │
│   ┌───────────────┐                                 │
│   │  VPE 모듈     │  ← IDEFICS2-8B (Fine-tuned)    │
│   │  (Visual      │                                 │
│   │  Preference   │                                 │
│   │  Extractor)   │                                 │
│   └───────┬───────┘                                 │
│           │                                         │
│     VP+ (좋아하는 속성) & VP- (싫어하는 속성)       │
│           │                                         │
│           ▼                                         │
│   ┌───────────────┐   [텍스트 프롬프트]             │
│   │ CLIP Text     │◄──────────────────              │
│   │ Encoder       │                                 │
│   └───────┬───────┘                                 │
│           │ e_cond = e_prompt ⊕ α·e_VP+ ⊖ β·e_VP-  │
│           ▼                                         │
│   ┌───────────────┐                                 │
│   │ Stable        │  ← No additional fine-tuning    │
│   │ Diffusion XL  │                                 │
│   │ (SDXL)        │                                 │
│   └───────┬───────┘                                 │
│           │                                         │
│    [개인화된 이미지 출력]                           │
└─────────────────────────────────────────────────────┘
```

Stable Diffusion을 추가 파인튜닝 없이 시각적 선호도에 따라 조건화할 수 있음을 보여준다.

### 2-4. 성능 향상

20명의 사용자를 연구에 초대하여 두 가지 유형의 테스트를 수행하였다. 각 참가자는 먼저 8~20개의 이미지(유연하게 선택 가능)에 대해 코멘트를 작성하도록 요청받았다. 그런 다음 각 사용자는 동일한 프롬프트로 생성된 8개의 이미지를 보여주었으며, 그 중 하나만 ViPer로 개인화된 것이었다. 이 과정은 15개의 프롬프트에 대해 반복되었다.

**테스트 1 (ViPer vs. No Personalization)**:
상위 1위 정확도(Top-1 Accuracy)가 **86.1%**로, 사용자가 가장 선호하는 이미지가 압도적으로 자신의 시각적 선호도로부터 생성되었으며, 무작위 선택보다 **6.89배** 높은 수준을 보였다.

**테스트 2 (ViPer vs. Other Users' Personalization)**:
다른 사용자의 개인화 이미지와 비교한 두 번째 테스트에서 상위 1위 정확도는 **65.4%**로, 무작위 선택보다 **5.23배** 높았다. 두 번째 테스트에서의 낮은 정확도는 일부 사용자가 유사한 선호도를 가지기 때문이다.

| 테스트 | Top-1 정확도 | 무작위 대비 배율 |
|---|---|---|
| ViPer vs. No Personalization | **86.1%** | 6.89× |
| ViPer vs. Other Users' Pers. | **65.4%** | 5.23× |

### 2-5. 한계점

두 번째 테스트에서의 낮은 정확도는 일부 사용자가 유사한 선호도를 가지기 때문이며, 이를 극복하려는 노력에도 불구하고 사용자들이 자신의 코멘트에 명시적으로 언급하지 않은 다른 사용자의 결과 세부 사항을 선호할 수 있다.

추가적인 한계점을 정리하면:

1. **코멘트 품질 의존성**: 더 자세하고 표현적인 코멘트가 더 나은 개인화로 이어진다는 점에서, 간결하거나 모호한 코멘트는 개인화 품질을 저하시킬 수 있음
2. **언어 의존성**: 코멘트 기반 방식으로 인해 언어 표현이 어려운 사용자에게는 불리함
3. **도메인 제한**: Stable Diffusion XL 기반 아키텍처에 한정되며 다른 생성 모델로의 직접 이식이 불명확함
4. **시각적 취향의 복잡성**: 시각적 취향은 단순히 하나의 예술 스타일이나 색상 팔레트를 좋아하는 것 이상으로, 개인적 선호도에 기반한 다양한 시각적 속성에 의해 형성된다는 복잡성으로 인해 일부 선호가 누락될 수 있음

## 3. 모델의 일반화 성능 향상 가능성

### 3-1. 일반화를 가능하게 하는 핵심 설계

코멘트들은 시각적 속성을 포함하는 구조화된 표현(예: surrealism, rough brushstrokes, monochromatic 등)으로 변환되며, Stable Diffusion을 추가 파인튜닝 없이 이러한 시각적 선호도에 조건화할 수 있다.

이는 곧 **파인튜닝 불필요(Fine-tuning Free)** 접근 방식이 일반화에 유리함을 시사한다:

- 특정 사용자나 도메인에 오버피팅될 위험이 없음
- 새로운 사용자 추가 시 모델 재학습 없이 VPE 추론만으로 적용 가능
- 다양한 프롬프트에 대해 학습 없이 즉시 개인화 가능

### 3-2. 시뮬레이션 에이전트를 활용한 일반화

프록시 측도를 얻기 위해 에이전트의 데이터와 Pick-a-Pic 데이터셋 두 가지 소스의 훈련 데이터를 결합하며, 에이전트의 시각적 선호도를 사용하여 ViPer로 각 에이전트에 대한 소규모 개인화 이미지를 생성한다. 추가로, 언어 모델로 개인화된 프롬프트를 생성하고 표준 Stable Diffusion XL을 사용하여 이미지를 생성한다. 이는 프록시 측도가 ViPer의 생성물에 편향되지 않도록 보장하기 위한 것이다.

이 **이중 소스 훈련(Dual-source Training)**은 다음과 같은 일반화 효과를 낸다:

$$
\mathcal{D}_{train} = \mathcal{D}_{agents} \cup \mathcal{D}_{Pick\text{-}a\text{-}Pic}
$$

- $\mathcal{D}_{agents}$: 시뮬레이션 에이전트 기반 선호 데이터 → 다양한 개인 특성 커버
- $\mathcal{D}_{Pick\text{-}a\text{-}Pic}$: 실제 인간 사용자의 이진 선호 데이터 → 실세계 일반화

### 3-3. 일반화 가능성의 방향

| 일반화 방향 | 가능성 | 설명 |
|---|---|---|
| **새로운 사용자** | ✅ 높음 | VPE 추론만으로 신규 사용자 온보딩 가능 |
| **새로운 프롬프트** | ✅ 높음 | 구조화된 속성이 프롬프트에 독립적으로 적용됨 |
| **다른 생성 모델** | ⚠️ 중간 | 프롬프트 임베딩 기반이므로 CLIP 계열 모델에 적용 가능 |
| **다른 도메인(비예술)** | ⚠️ 중간 | 현재는 시각 예술 스타일 중심; 다른 도메인 확장 필요 |
| **다국어 코멘트** | ❓ 미확인 | VPE의 다국어 처리 성능이 검증되지 않음 |

## 4. 앞으로의 연구에 미치는 영향 및 고려할 점

### 4-1. 연구에 미치는 영향

#### 🔷 인간-AI 상호작용(Human-AI Interaction) 패러다임 변화

ViPer는 **"사용자가 모델에 적응"하는 패러다임**에서 **"모델이 사용자에게 적응"하는 패러다임**으로의 전환을 보여준다. 이는:
- 생성 AI의 대중화와 접근성 향상
- 비전문가 사용자도 깊은 기술 지식 없이 원하는 결과물 생성 가능

#### 🔷 개인화된 RLHF (Reinforcement Learning from Human Feedback) 방향성 제시

기존 RLHF는 집계된(aggregated) 인간 선호도를 사용하지만, ViPer는 **개인 수준(individual-level)**의 선호 학습을 보여줌으로써:
- 개인화된 reward model 설계 연구 촉진
- 다양한 사용자 집단을 포괄하는 공정한 AI 생성 연구에 기여

#### 🔷 VPE의 확장 가능성

VPE는 이미지 집합과 사용자의 코멘트에서 개인적 선호도를 추출하는 파인튜닝된 비전-언어 모델로, 이미지 생성 외에도 추천 시스템, 디지털 콘텐츠 큐레이션 등 다양한 분야에 응용될 수 있다.

#### 🔷 평가 지표 혁신

기존의 FID, CLIP Score 같은 집계 지표 대신 **개인화된 프록시 평가 지표(Personalized Proxy Metric)** 라는 새로운 평가 패러다임을 제시하였다.

---

### 4-2. 앞으로 연구 시 고려할 점

#### ① 선호의 시간적 변화(Temporal Drift) 처리
사용자의 선호는 시간이 지남에 따라 변화한다. **지속적 학습(Continual Learning)** 또는 **온라인 업데이트** 메커니즘 통합이 필요하다:

$$
\mathbf{VP}^+(t) = f\left(\mathbf{VP}^+(t-1), \Delta\mathcal{C}^+(t)\right)
$$

여기서 $\Delta\mathcal{C}^+(t)$는 시간 $t$에서의 새로운 코멘트.

#### ② 다양한 문화적 배경 및 언어의 일반화
현재 VPE는 영어 코멘트 기반으로 설계되었을 가능성이 크며, 다국어 및 다문화 환경에서의 성능 검증이 필요하다.

#### ③ 프라이버시와 선호 데이터 보호
사용자의 시각적 선호도는 민감한 개인 정보일 수 있으므로:
- **Federated Learning** 적용 가능성
- 차분 프라이버시(Differential Privacy) 기법 통합

#### ④ 멀티모달 확장
현재는 텍스트 코멘트만 수집하지만, 향후에는:
- 사용자가 직접 그린 스케치
- 음성 코멘트
- 시선 추적(eye-tracking) 데이터 등 멀티모달 입력 통합 연구

#### ⑤ 생성 모델 다양화
현재는 Stable Diffusion을 조건화하는 방식이나, DiT(Diffusion Transformers), Flux, DALL-E 3 등 다른 최신 생성 모델로의 확장 연구가 필요하다.

## 5. 2020년 이후 관련 최신 연구 비교 분석

| 논문 | 연도 | 방법론 | 개인화 방식 | 파인튜닝 필요 | 한계 |
|---|---|---|---|---|---|
| **Textual Inversion** (Gal et al.) | 2022 | 텍스트 임베딩 최적화 | 특정 개념 학습 | ✅ 필요 | 새 개념마다 재학습 |
| **DreamBooth** (Ruiz et al.) | 2023 | 모델 파인튜닝 | 피사체 중심 | ✅ 필요 | 비용 높음, 오버피팅 위험 |
| **FABRIC** (von Rütte et al.) | 2023 | 반복적 피드백 | 이미지 선택 기반 | ❌ 불필요 | 매 생성마다 반복 피드백 필요 |
| **InstructPix2Pix** (Brooks et al.) | 2023 | 편집 지시 학습 | 텍스트 명령 기반 | ❌ 불필요 | 스타일 개인화보다 편집에 집중 |
| **Pick-a-Pic** (Kirstain et al.) | 2023 | 이진 선호 데이터 | 집계된 선호 | ✅ 필요 | 개인 수준 아닌 집단 수준 |
| **ViPer** (Salehi et al.) | 2024 | VPE + 프롬프트 임베딩 | 개인 코멘트 기반 | ❌ 불필요 | 코멘트 품질 의존 |
| **DPG-T2I** (ECCV 2024) | 2024 | 강화학습 기반 | 개인화된 컬렉션 | ✅ 필요 | 강화학습의 높은 계산비용 |

### 핵심 차별점

ViPer는 생성 모델의 출력을 서로 다른 사용자의 시각적 선호도에 맞게 개인화하는 방법으로, 사용자의 일반적인 선호도를 일회성으로 캡처하고 상세한 프롬프트를 엔지니어링할 필요 없이 생성 모델을 조건화한다.

이는 DreamBooth나 Textual Inversion처럼 **매번 파인튜닝이 필요한 방식**과 달리, 한 번의 선호 수집만으로 무한한 프롬프트에 적용 가능하다는 점에서 실용성이 매우 높다.
